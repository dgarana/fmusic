## 0.1.0

### Bug Fixes

- Fix: play tracks with non-ASCII titles in dev and prod
Two related bugs made some tracks unplayable:
1. Titles with non-ASCII characters (e.g. 'Tïesto') came back from yt-dlp
   decoded through the Windows console code page, not UTF-8. The stored
   file path then did not match the file on disk and trackStreamUrl
   silently returned null.
   - ytdlp.ts: all spawns now run with PYTHONIOENCODING=utf-8,
     PYTHONUTF8=1 and LC_ALL/LANG=C.UTF-8 so stdout is always UTF-8.
   - Every chunk.toString() is replaced with an explicit toString('utf8').
   - The filepath emitted by --print is normalised with
     path.normalize().normalize('NFC') to guard against NFD/NFC drift.
   - tracks-repo.ts: new resolveTrackFilePath() self-heals old, mis-encoded
     DB rows by scanning the folder for a file containing [youtubeId] and
     patching file_path. Used by both the stream and delete handlers.
2. npm run dev served the renderer from http://localhost:<port>, which
   made Chromium block <audio src="file://..."> on cross-origin grounds.
   The packaged build worked because it loads from file://, so the bug
   only showed up in dev.
   - main/index.ts: register a privileged fmusic-media:// scheme and a
     protocol.handle that maps fmusic-media://track/<id> to the track's
     local file via net.fetch(pathToFileURL(...)). Range requests for
     seeking keep working because Electron's net stack handles them.
   - ipc.ts: tracks:stream-url now returns fmusic-media://track/<id>
     instead of a file:// URL.
   - CSP (both main and index.html) allows fmusic-media: in media-src
     and connect-src. ([d597d69](https://github.com/dgarana/fmusic/commit/d597d696ba03a1a9dfc9a37ad04edc976ce636fe))

### Features

- Feat: initial scaffold of fmusic Electron app
Multiplatform desktop app (Windows, macOS, Linux) for downloading music from
YouTube via yt-dlp, managing a local SQLite library, and playing it back.
Stack:
- Electron 33 (main + preload + renderer, contextIsolation)
- Vite + React 18 + TypeScript in the renderer
- electron-vite bundler, electron-builder for distributables
- better-sqlite3 with forward-only migrations (PRAGMA user_version)
- Zustand stores; Howler for audio playback
- yt-dlp and ffmpeg shipped as platform-specific binaries via postinstall
Includes:
- Download (URL + YouTube search + iframe preview + progress queue)
- Library (sortable table, search, genre filter)
- Playlists (create, add/remove/reorder tracks)
- Settings (download folder, format/quality, dependency status,
  in-app yt-dlp updater, schema history)
- GitHub Actions release workflow for Windows/macOS/Linux ([71da378](https://github.com/dgarana/fmusic/commit/71da37834032434c94f9559b6a251e0bf00ca3a8))
- Feat: rich YouTube metadata, inline audio preview, playlist picker, badges
- Preview: replace YouTube iframe (blocked on file:// origins) with a direct
  audio stream resolved via 'yt-dlp -f bestaudio -g'. CSP widened to allow
  media-src https://*.googlevideo.com / *.youtube.com. Fallback to opening
  the video in the default browser when the stream cannot be resolved.
- Metadata: extract artist/album/track/genre/release_year from yt-dlp's
  --dump-json (artist/creator/uploader cascade, genres/categories). Five
  '--parse-metadata' directives now embed these fields as ID3 tags inside
  the downloaded MP3 so other players pick them up. DownloadManager
  prefers yt-dlp-resolved metadata and falls back to music-metadata ID3
  reads only when a field is missing.
- Playlist detail: new '+ Añadir canciones' picker with search filter and
  multi-select checkboxes over library tracks not already in the playlist.
  Bulk addTrackToPlaylist on confirm.
- Library table: new 'Playlists' column rendering pill badges per track.
  Backed by a new playlistsForTracks(ids[]) IPC that batches the lookup
  into a single 'WHERE track_id IN (?)' query. ([1753604](https://github.com/dgarana/fmusic/commit/1753604cd2631a6ebddedd41a180721e1f68dc42))
- Feat: polish download UX + fix player replay after queue ends
Download page:
- Cards now carry their own state: progress bar + status-pill when a job
  matches the result, '✓ En biblioteca' (disabled) when the youtubeId is
  already in the DB, 'Cancelar' while active, normal 'Descargar' otherwise.
- Preview is now inline per-card and mutually exclusive: clicking
  'Previsualizar' replaces the button with 'Cerrar preview' and mounts an
  <audio> element inside that card; switching to another card tears the
  previous preview down automatically.
- Search query + results are persisted in a new Zustand store
  (src/renderer/src/store/search.ts) so navigating away and back keeps the
  results visible instead of resetting to empty.
- Jobs not matching any visible result (e.g. direct URL pastes) are shown
  under 'Otras descargas' at the bottom.
Backend:
- DownloadJob now carries youtubeId (set from the 'info' event).
- New findDownloadedYoutubeIds(ids[]) repo helper + IPC channel
  tracks:downloaded-ids for batched library membership lookup.
- Added extractYoutubeId(url) util to link jobs to results before yt-dlp
  resolves metadata.
Player fixes:
- Thumbnails: PlayerBar falls back to
  https://i.ytimg.com/vi/<id>/hqdefault.jpg when thumbnailPath is null,
  so covers show up for every existing track.
